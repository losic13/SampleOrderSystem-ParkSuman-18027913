# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 배경

가상의 반도체 회사 "S-Semi"는 다양한 반도체 시료(Sample)를 생산해 연구소·팹리스 업체·대학 연구실에
납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고 검수를 거쳐 출고된다. 기존에는
엑셀/메모장으로 주문을 관리해 실수가 잦고 재고·공정 현황을 한눈에 파악하기 어려웠다. 이를 해결하기 위한
콘솔 기반 "반도체 시료 생산주문관리 시스템"이 이 문서의 대상이다.

이 저장소는 5개로 나뉜 과제 저장소 중 마지막(미션2, 최종 통합 시스템)이며, 이전 4개 PoC —
ConsoleMVC(MVC 골격), DataPersistence(JSON 영속성), DataMonitor(모니터링), DummyDataGenerator(더미데이터)
— 에서 각각 검증한 설계를 이어받아 하나의 완성된 시스템으로 재구현한다.

## 2. 역할

| 역할 | 책임 |
|---|---|
| 고객 (시료 요청자) | 필요한 시료를 이메일 등으로 요청 (시스템 외부 행위) |
| 주문 담당자 | 요청에 맞게 주문서 작성(주문 접수), 출고 처리 |
| 생산 담당자 | 시료 등록, 주문 승인/거절, 생산라인 관리 |

이 시스템은 콘솔 기반으로 동작하며, 담당자가 직접 명령을 입력해 시료를 등록하고 주문을 처리한다.

## 3. 용어 정의

- **시료(Sample)**: 시스템의 가장 기본 단위. 고유 ID·이름·평균 생산시간·수율·재고를 가진다.
  등록된 시료만 주문 가능하다.
- **수율(yield_rate)**: 정상적으로 생산된 시료 비율. `정상 생산 시료 수 / 총 생산 시료 수`
  (예: 100개 생산 중 정상 90개 → 0.9).
- **생산라인**: 시료를 하나씩 생산하는 단일 설비 흐름. 주문이 들어온 시료에 대해서만 생산하며,
  대기 작업은 FIFO(선입선출) 큐로 관리한다.

## 4. 주문 상태 흐름

| 상태 | 의미 |
|---|---|
| `RESERVED` | 주문 접수 |
| `REJECTED` | 주문 거절 (정상 흐름 밖, 모니터링 대상 제외) |
| `PRODUCING` | 승인 완료 + 재고 부족으로 생산 중 |
| `CONFIRMED` | 승인 완료 + 재고 확보, 출고 대기 중 |
| `RELEASE` | 출고 완료 |

```
RESERVED --(승인, 재고충분)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(승인, 재고부족)--> PRODUCING --(생산완료)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(거절)--> REJECTED
```

## 5. 기능 명세 (메뉴별)

### [1] 시료 관리
- 시료 등록: 시료 ID, 이름, 평균 생산시간, 수율, (선택)초기 재고 입력
- 시료 목록 조회: 등록된 전체 시료와 현재 재고 표시
- 시료 검색: 이름 키워드로 검색 (대소문자 무시, 부분일치)

### [2] 시료 주문
- 고객이 원하는 시료 ID·고객명·주문 수량을 입력받아 `RESERVED` 주문 생성
- 등록되지 않은 시료 ID, 1 미만의 수량은 거부

### [3] 주문 승인/거절
- `RESERVED` 상태 주문 목록 표시
- 승인: 재고 확인 후
  - 재고 충분 → 즉시 `CONFIRMED`, 재고에서 주문 수량 차감
  - 재고 부족 → `PRODUCING`, 생산라인 큐에 등록 (아래 §7 참고)
- 거절: 즉시 `REJECTED`

### [4] 모니터링
- 주문량 확인: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 수 (`REJECTED` 제외)
- 재고량 확인: 시료별 현재 재고 및 상태(여유/부족/고갈) — 판정 기준은 §7 참고

### [5] 생산라인 조회
- 현재 처리 중인 생산 작업(주문 정보, 실생산량, 진행률) 표시
- 대기 중인 생산 큐를 FIFO 순서로 표시
- `[T] 시간 경과`: 사용자가 지정한 분(minute)만큼 진행 (수동 tick 시뮬레이션, §7 참고)

### [6] 출고 처리
- `CONFIRMED` 상태 주문 목록 표시, 선택한 주문을 `RELEASE`로 전환

## 6. 데이터 모델

| 엔티티 | 필드 |
|---|---|
| Sample | sample_id, name, avg_production_time(분/개), yield_rate(0~1), stock |
| Order | order_id, sample_id, customer_name, quantity, status, created_at |
| ProductionJob | order_id, sample_id, shortage, actual_quantity, total_minutes, elapsed_minutes, enqueued_at |

## 7. 생산라인 규칙 (핵심 설계 결정)

- **실생산량** = `ceil(부족분 / 수율)`
- **총 생산시간** = `평균생산시간 × 실생산량`
- **스케줄링**: FIFO, 생산라인은 1개(동시에 하나의 작업만 진행)
- **시간 진행**: 실제 시계가 아닌 **수동 tick** — 사용자가 진행시킬 분(minute)을 직접 입력해야 진행된다.
  결정적(deterministic)이라 unit-test로 검증하기 쉽다.
- **재고 예약 모델 (중요)**: 승인 시 재고가 부족해 `PRODUCING`으로 전환하는 순간, 그 시점의 가용 재고를
  즉시 0으로 차감한다 (재고가 충분할 때 `CONFIRMED`가 즉시 재고를 차감하는 것과 동일한 원칙 — "승인된
  주문은 그 순간 재고를 확정 소비한다"). 이렇게 하지 않으면 같은 시료에 대한 다른 주문이 그 사이 동일한
  재고를 또 소비하는 이중 예약 버그가 생긴다. 생산이 완료되면 `sample.stock += shortage`로 부족분만큼만
  재고를 채우고 주문을 `CONFIRMED`로 전환한다.
- **가정/비목표**: 수율에 따른 초과 생산분·폐기분의 재고 반영은 다루지 않는다 (부족분만큼 정확히
  생산된다고 가정). 생산라인은 1개만 존재하며 병렬 생산은 다루지 않는다. 실시간(wall-clock) 타이머,
  동시성(멀티스레드) 처리는 범위 밖이다.

## 8. 모니터링 재고 상태 판정

- **고갈**: 재고 0
- **부족**: 재고 < `RESERVED` 상태 주문 수량 합 (미해결 수요)
- **여유**: 그 외

> DataMonitor PoC는 `PRODUCING` 주문도 미해결 수요에 포함했지만, 이 시스템은 `PRODUCING` 전환 시 재고를
> 이미 0으로 예약 처리하므로 `RESERVED`만 미해결 수요로 계산한다 (자세한 이유는 CLAUDE.md 참고).

## 9. 아키텍처 (요약)

레이어드 구조: `model` → `repository`(JSON 영속성) → `controller`(비즈니스 로직) → `view`(콘솔 UI),
`monitor`(조회 전용 집계). 상세 설계와 각 결정의 이유는 `CLAUDE.md`를 참고한다.

## 10. 데이터 저장

JSON 파일 (`data/samples.json`, `data/orders.json`, `data/production_jobs.json`). 프로세스를 재시작해도
데이터가 유지된다.

## 11. 품질 기준 (Harness)

- 전 계층 unit test (pytest), 종단 시나리오 테스트 1개
- `pytest-cov`로 커버리지 측정
- 프로젝트 전용 `/verify` 스킬로 태스크 완료 시점마다 테스트 + 콘솔 앱 스모크 실행 확인
- 단위테스트가 딸린 태스크(모델/저장소/컨트롤러/생산라인/모니터링/종단테스트)는 **Red-Green-Review**
  3단계로 커밋을 나눈다 (실패 테스트 → 최소 구현 → 리팩터링). 뷰/진입점, 문서화 태스크는 1커밋 유지
- 코드 품질 검토는 저장소 전용 스킬 `architecture-guardian`(OCP/SOLID + MVC 레이어링 + 재고 예약
  모델 불변식 검토, REVIEW 단계에서 사용) + 전역 `code-review`/`simplify`(버그·중복·효율성) 조합으로
  수행하고, 진행 상황은 네이티브 Task 도구로 추적한다. 상세 근거는 `CLAUDE.md`의 "Harness" 섹션 참고
- 태스크 단위로 커밋 (스캐폴딩 → 하네스 구축 → 모델 → 저장소 → 컨트롤러 → 생산라인 → 모니터링 →
  뷰/진입점 → 종단테스트 → 문서화)
